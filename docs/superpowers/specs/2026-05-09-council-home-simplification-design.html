D.R.A.M.A. Design Spec

Council Home Simplification

Replace the current landing-page-style shell with a prompt-first home screen: one shared box for typing or speaking, one Ask Council action, and a visible row of council member photos that become directly talkable once the council is ready.

Date: 2026-05-09 Scope: Home screen redesign Mode: In-place state transitions

Goal

  • Open directly into a focused product surface instead of a demo or marketing page.
  • Support typing and voice-to-text through one shared prompt box.
  • Keep the council visually present with a compact row of member photos.
  • Let the user tap a specific council member photo after submit to talk live with that member.

Non-Goals

  • Changing backend Friends Mode initialization behavior.
  • Redesigning the downstream live voice runtime.
  • Adding onboarding, settings, or extra marketing sections.
  • Keeping the current hero, architecture, transcript, or workspace panels in the first-view experience.

Experience Summary

The app opens to one screen. At the top is a small visible row of five council member photos. Beneath that is one prompt composer with a mic control and one primary Ask Council button. After submit, the same screen transforms in place from idle to thinking to ready. Once ready, each member photo can be tapped to start a live conversation with that specific member.

Visual Direction

Chosen Tone

Use the earlier B structure with the restraint of C: council presence up top, almost no intro copy, and no landing-page energy.

Personality Source

Keep personality in the photos and the live-state behavior, not in marketing copy, loud gradients, or oversized hero composition.

Primary Flow

  1. User lands on the home screen and sees the council photo row, one prompt box, and one primary button.
  2. User types or taps the mic to speak. Voice transcription populates the same prompt box.
  3. User presses Ask Council.
  4. The screen stays in place and the council enters a visible thinking state.
  5. When ready, the photos become the main interaction targets.
  6. User taps a member photo and starts talking directly with that member.

Component Structure

Keep

  • Existing Friends Mode runtime logic in src/App.tsx.
  • Current browser speech recognition flow.
  • Current live voice preview startup and cleanup behavior.

Refactor Toward

  • CouncilHome
  • CouncilPhotoRow
  • PromptComposer
  • Smaller focused state rendering instead of one large landing page shell

Interaction Rules

Voice Input

  • The mic belongs to the prompt composer, not a separate screen.
  • Speaking appends recognized text into the same prompt box.
  • If unsupported, typed input still fully works.

Council Photos

  • Use photos, not initials or generic badges.
  • Show all five before submit.
  • Only ready members are tappable.
  • The active member shows a clear live state.

State Model

  • idle: no submitted question yet
  • listening: browser speech recognition is active
  • thinking: question submitted, council preparing
  • ready: members can be tapped
  • live: one member is currently active for voice conversation

Error Handling

  • Keep errors inline and short.
  • Preserve the typed prompt on failures.
  • Cover empty prompt, speech recognition failure, initialization failure, and preview startup failure.

Testing Focus

  • Typed and spoken input both feed the same prompt field.
  • Ask Council transitions cleanly through thinking and ready states.
  • Member photos are visible before submit and tappable after ready.
  • Tapping a ready member starts that member's live session.
  • The old landing-page sections are gone from the first-view experience.

Recommendation

Implement this as a true simplification refactor, not a new layer on top of the current page. The product already has the right behavior primitives. The work is mainly to strip away the landing-page framing, restructure the UI into a single calm screen, and replace abstract avatar treatments with real council member photos.
