The product feels like two features, but the implementation is one reusable council session with different persona packs.

Onboard The Bias

D.R.A.M.A. works because every advisor is biased on purpose. The onboarding makes those biases explicit before any session starts.

Persona Setup

Each avatar gets a name, mode membership, personality summary, response style, blind spots, repeated phrases, and boundaries.

Personality Response style Blind spots Notes

Formative Backstory

The user writes or fictionalizes what made this advisor this way, so every answer carries a visible point of view.

No inferred private history Bias is the feature

Fast vs Pro

Fast is the live judging path. Pro is the deeper research path that makes the product feel serious after the laugh lands.

Fast D.R.A.M.A.

Immediate realtime voice feedback. The council gives short takes, one disagreement round, visible votes, and a verdict while the user is still in the room emotionally.

Realtime voice No external research Best for demo

Pro D.R.A.M.A.

Starts with the same voice intake, then runs background research. Exa pulls receipts; Manus can run a deeper async advisor task. The output is a case file and follow-up summary.

Receipts Case file Async research

Voice Demo Flow

Keep the judging flow short, theatrical, and resilient to network or microphone failure.

1 Set the council Use onboarding or defaults to define each advisor's personality and bias.
2 Pick chamber and tier Choose Inner or Group, then Fast for live or Pro for research.
3 Speak the dilemma User starts voice, asks a decision or pitches an idea.
4 Council debates The realtime moderator performs short persona turns with speaker markers.
5 Verdict lands Fast shows a verdict. Pro queues receipts and returns a case file.

Architecture

One OpenAI Realtime moderator session is the MVP path. Separate per-persona agents can come after the live demo works.

Frontend

  • React + Vite + TypeScript.
  • Onboarding, mode selector, tier selector, and shared council room.
  • Persona panels with active speaker state.
  • Transcript, research, and verdict panels.
  • Text fallback plus replay demo trace.
Browser WebRTC
plus data-channel events

Backend

  • Node + Express.
  • GET /api/realtime-token?mode=inner|group
  • POST /api/pro-research for Pro case files.
  • Server keeps OPENAI_API_KEY private.
  • Returns only ephemeral Realtime token payload.

Non-Goals

These are intentionally out of scope until the voice council is stable enough to demo.

No friend uploads

Real photo-based avatars need consent, image processing, and extra risk.

No inferred biography

Backstories must be user-written or fictional, not guessed from photos or names.

No provider router

OpenAI Realtime first. Everything else stays optional or Pro-only.
